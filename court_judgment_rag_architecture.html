Data layer — storage architecture

Tiered hot/cold strategy. Parquet for queryable metadata; pgvector for live similarity search. No need to load cold embeddings unless retrieved.

Scale estimate: SC ~3M judgments (all years), HC ~15M judgments (all states). Chunked at 512 tokens, 128 overlap → roughly 80–120M vectors in hot store. Use pgvector with HNSW (not IVFFlat) at this scale — HNSW gives better recall without full scan.

Lightweight metadata extraction — no LLM required

The challenge: extract area_of_law, case_type, and outcome for millions of documents cheaply. Solution: a three-layer cascade of increasing cost.

Key insight: for many fields, metadata IS NOT needed for quality retrieval. Use offline clustering (k-means on embeddings) as a semantic pre-filter instead of explicit labels. Clustering naturally separates service law, property law, criminal matters, etc. without any manual annotation.
Layer 1 — regex patterns

Extracts 80% of metadata at near-zero cost.

• Case number (all court formats)

• Date of order (multiple date formats)

• Court name (finite dictionary of ~50 courts)

• Presiding judge names

• Citation patterns (SCC, AIR, SC, HC)

• Disposition words (dismissed/allowed/partly)

• Petition type from case number prefix

Speed: ~50,000 docs/min on single CPU

Layer 2 — DistilBERT classifier

Fine-tuned on ILDC dataset (Indian Legal Documents Corpus, ~35k labeled cases).

• Area of law (47 categories)

• Case type (writ/appeal/revision/SLP)

• Document type (judgment/order/interim)

Model: DistilBERT (66M params) — 4× faster than BERT, 97% of accuracy. Input: first 512 tokens of judgment (preamble + headnote).

Speed: ~3,000 docs/min on A100 GPU

Training data: ILDC + LEGAL-BERT fine-tune

Layer 3 — cluster assignment

Backup when classifier is uncertain (confidence <0.7).

• Embed first 512 tokens with InLegalBERT

• Assign nearest cluster_id from weekly k-means centroids

• cluster_id acts as a soft metadata substitute — cases in same cluster share legal domain even without explicit labels

k=500 clusters gives fine-grained topic separation without explicit labeling.

Citation graph — building appeal lineage

Build a directed graph: if case B's text cites case A (extracted via regex), draw edge A→B. If B "reversed" or "set aside" A (keyword detection), mark edge as appeal_overturned. This graph lets you look up whether any HC judgment was subsequently appealed to SC — without an LLM on any document.

citation_edges( from_case_id STRING, -- case that was appealed to_case_id STRING, -- appellate judgment edge_type ENUM, -- 'appeal', 'reference', 'relied_on' outcome ENUM -- 'reversed', 'confirmed', 'modified' ) -- Index: from_case_id → fast "was this appealed?" lookup

Retrieval pipeline — hybrid search with semantic safety

Five-stage funnel: metadata pre-filter → BM25 shortlist → dense retrieval → RRF fusion → cross-encoder reranking. Each stage shrinks the candidate pool while improving relevance.

The core challenge you identified — preventing semantic contamination across unrelated legal domains — is solved by cluster-gated retrieval: restrict search to the same cluster(s) as the query before running any similarity computation.
1
Query preparation — from your extracted JSON
Your extraction pipeline already gives you area_law, case_type, primary_statute, court_directions, citations. Build two query forms:
Lexical query: statute names + legal issue keywords + court direction verbs → BM25
Semantic query: embed the ratio_decidendi + issues_framed (most discriminative sections)
Also extract cluster_id by embedding ratio_decidendi and finding nearest k-means centroid.
2
Stage 1 — metadata pre-filter (if available)
SQL filter on Parquet / pgvector metadata table before any vector ops:
WHERE cluster_id IN (query_cluster, nearest_2_clusters) AND date_of_order >= NOW() - INTERVAL '10 years' -- hot store default AND court_code IN ('SC', same_HC_code) -- jurisdiction filter AND (area_law_id = query_area_law OR area_law_id IS NULL) -- If area_law_id missing → cluster_id alone provides isolation
This reduces search space from 40M → ~100K-500K cases before any embedding comparison.
3
Stage 2 — hybrid retrieval with RRF fusion
Run BM25 + dense retrieval in parallel on the pre-filtered set, then fuse with Reciprocal Rank Fusion (not weighted sum — RRF is more robust to score scale differences).
-- Dense (pgvector) SELECT case_id, 1 - (embedding <=> query_vec) AS dense_score FROM case_chunks WHERE cluster_id = ANY(...) ORDER BY embedding <=> query_vec LIMIT 200; -- BM25 (postgres full-text) SELECT case_id, ts_rank(fts_vector, query_tsquery) AS bm25_score FROM case_chunks WHERE cluster_id = ANY(...) LIMIT 200; -- RRF: score = 1/(60+rank_dense) + 1/(60+rank_bm25) -- Take top 100 by RRF score
4
Stage 3 — parent document aggregation
Chunks point back to parent cases. Aggregate chunk scores per case: use max(chunk_score) per case, not sum — you want the single most-relevant passage to determine case relevance. Deduplicate to get top 50 unique cases.
5
Stage 4 — cross-encoder reranking → top 20
Cross-encoder takes (query_text, candidate_text) pairs and produces a more accurate relevance score than embedding cosine similarity. Query text = ratio_decidendi + issues. Candidate text = top-3 chunks from each case.
Model: cross-encoder/ms-marco-MiniLM-L-12-v2 fine-tuned on Indian legal pairs (or use as-is — generalises well). Reduces top-50 → top-20 with significantly better precision.
Cost: 50 forward passes, fast at inference (~2s on GPU).
6
Stage 5 — diversity and appeal-lineage filter
Final post-processing before agent input:
• MMR (Maximal Marginal Relevance) — ensure top-20 aren't all from the same sub-statute or same year
• Boost cases where is_appealed=true and appeal_outcome is known — most valuable for comply/appeal analysis
• Boost SC cases over HC cases (more authoritative precedent)
Output: top-20 cases with appeal lineage, win/loss history, and top-3 relevant chunks each
How cluster-gating prevents cross-domain contamination
A gratuity dispute and a land acquisition case might share words like "government", "order", "directions", "compliance" — these would score high on BM25 and even moderately on dense retrieval without domain isolation.

Cluster-gating (k=500 clusters) naturally separates these: the embedding space for service law cases clusters far from property law cases. By restricting Stage 1 to the query's nearest 3 clusters, you ensure all 200 BM25 + dense candidates are semantically proximate to the query domain — even if individual cases lack explicit area_of_law labels.

This is better than hard metadata filtering because: (a) it works even when labels are missing, (b) it handles edge cases and multi-domain cases naturally, and (c) cluster boundaries are learned from actual case content, not a rigid taxonomy.

Agent 1 — Historical Research Agent parallel
Input: top-20 case texts + metadata
Llama 3.3 70B
For each of the 20 similar cases, extract: key legal arguments used, statutes relied on, key holdings, whether the government was petitioner or respondent, what the final outcome was, and (from citation graph) whether it was further appealed and the outcome. Produces a structured per-case research report. This is a map operation — each case processed in a single prompt, results collated.
Output per case: {case_id, arguments_for_government, arguments_against_government, key_holdings[], statutes_cited[], government_role, outcome, was_appealed, appeal_outcome, distinguishing_factors[]}
Rules Engine — Statistical Analyzer parallel
Input: appeal_outcome fields from top-20 + current case metadata
deterministic Python
NOT an LLM. Pure Python calculations. Produces hard numbers that agents cannot reliably compute:
win_rate_on_appeal = count(appeal_outcome='allowed') / count(is_appealed=True) lose_rate_on_appeal = count(appeal_outcome='dismissed') / count(is_appealed=True) limitation_deadline = date_of_order + statute_limitation_period(appeal_type) # Service law HC→DB: 30 days; HC→SC SLP: 90 days; etc. days_remaining = limitation_deadline - today() financial_exposure = sum(financial_implications[]) govt_as_petitioner_win_rate = ... (filtered by government_role=petitioner)
Output: hard stats dict passed to every subsequent agent as grounding context.
Agent 2 — Legal Argument Mapper
Input: Agent 1 output (all 20 case reports) + current case JSON
Llama 3.3 70B
Synthesizes across all 20 cases to build a structured argument landscape. Groups winning arguments (pro-appeal) and losing arguments (pro-comply) by legal principle. Maps citations that the current case could rely on vs. citations that cut against. Identifies which statutes/principles were consistently upheld vs. rejected in similar cases.
Output: { appeal_arguments: [{principle, cases_supporting[], citations[], strength}], comply_arguments: [{principle, cases_supporting[], citations[], strength}], current_case_citations_analysis: {supportive[], adverse[], neutral[]}, key_distinguishing_factors: [] // ways current case differs from similar }
Agent 3 — Adversarial Analysis Agent
Input: current case JSON + Agent 2 argument map + Rules Engine stats
Llama 3.3 70B
Plays devil's advocate in both directions. For appeal path: what are the realistic grounds? What's the probability of success given the win_rate and distinguishing factors? What are the procedural risks (limitation period, court fees, further delay)? For comply path: what are the risks of non-compliance (contempt, interest, further litigation)? Are there alternative routes (modification application, review petition, compliance with stay)?
Output: { appeal_viability: {score_0_to_10, grounds[], risks[], procedural_notes}, comply_viability: {score_0_to_10, compliance_steps[], contempt_risk, interest_cost}, alternative_routes: [{type, description, pros[], cons[]}], loopholes_in_judgment: [], strongest_appeal_ground: string, red_flags: [] // reasons that make appeal risky }
Agent 4 — Recommendation Synthesis Agent final
Input: ALL previous outputs (Agents 1-3 + Rules Engine)
Llama 3.1 405B (or Llama 3.3 70B)
Synthesizes a final verdict. Uses rules engine numbers as hard constraints (e.g., if days_remaining < 7, appeal urgency is CRITICAL regardless of merits). Uses agent scores as soft signals. Generates a structured recommendation with full traceable reasoning — every claim must cite specific cases or stats from prior agents.
Output: { verdict: "APPEAL" | "COMPLY" | "COMPLY_WITH_MODIFICATION" | "REVIEW_PETITION", confidence: float, // 0.0-1.0 urgency: "CRITICAL" | "HIGH" | "MEDIUM", deadline: date, primary_reasoning: string, // 3-5 sentence summary detailed_reasoning: { // full chain-of-thought statistical_basis: ..., // from rules engine precedent_analysis: ..., // from agents 1-2 risk_assessment: ..., // from agent 3 recommended_grounds: [] // if appeal: specific grounds to file }, citations_used: [], cases_relied_on: [], action_plan: [...] // merged with your directive extraction }

Technology stack recommendations

Best tools for each layer given your existing Nvidia NIM + Llama 3.3 70B + PostgreSQL setup.

Embeddings

Primary: law-ai/InLegalBERT

Fine-tuned on Indian legal corpus. 768-dim. Best recall on ILDC benchmark. Available on HuggingFace.

Alternative: nomic-embed-text-v1.5

8192 token context — can embed entire short judgments as single vector. Better for long-context retrieval.

Cross-encoder reranker

Primary: cross-encoder/ms-marco-MiniLM-L-12-v2

Fast, accurate on passage reranking. 6-layer MiniLM — 4× faster than BERT cross-encoder with minimal accuracy loss. Works well zero-shot on legal text.

Better (if time): Fine-tune on Indian legal QA pairs from ILDC/LegalLens dataset.

Vector database

You already have PostgreSQL. Add pgvector extension and use HNSW index.

CREATE INDEX ON case_chunks USING hnsw (embedding vector_cosine_ops) WITH (m=16, ef_construction=128);

At 40M vectors, pgvector HNSW is competitive with Pinecone/Qdrant and saves infra complexity.

LLM — agent pipeline

Agents 1-3: Llama 3.3 70B (Nvidia NIM)

128K context. 70B has strong instruction following for structured JSON output. Use NIM API for throughput.

Agent 4 (final): Llama 3.1 405B or Llama 3.3 70B

405B gives better reasoning chains but higher latency. For hackathon demo, 70B is sufficient.

Agent orchestration

Recommended: LangGraph

Supports parallel node execution (Agents 1 + Rules Engine run concurrently), conditional edges (cold storage fallback), and state persistence between nodes.

Alternative: CrewAI — simpler setup, less control over parallel execution.

Parquet + batch processing

Query engine: DuckDB

Query Parquet files directly from S3 without loading into memory. Sub-second aggregations on tens of millions of rows. Ideal for the pre-filter step.

Ingestion batch: Apache Spark or Ray Data for parallel PDF processing at scale.

Metadata classifier

Model: distilbert-base-uncased fine-tuned on ILDC

Dataset: ILDC (Indian Legal Documents Corpus) — 35k labeled SC cases with 9 categories. Fine-tune for area_of_law + case_type in ~2hrs on A100.

For hackathon: use a keyword-based heuristic as baseline, replace with fine-tuned classifier in production.

Offline clustering

Algorithm: k-means (k=500) via FAISS

FAISS k-means scales to 40M vectors efficiently. Run weekly as a batch job. Store centroids in PostgreSQL. For each new case, centroid assignment is a single nearest-neighbor lookup — microseconds.

Alternative: HDBSCAN for density-based clustering — finds natural cluster count but harder to assign new points.
